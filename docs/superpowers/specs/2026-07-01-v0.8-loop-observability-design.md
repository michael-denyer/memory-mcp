# MemoryMCP v0.8.0 — Trustworthy Loop: Observability and Utility Feedback

## Context

The Stop-hook learning loop (log response → mine patterns → promote memories) silently failed for five months: the shell hook passed `--session-id` to `run-mining`, which rejects it, and `log-response` dropped session provenance. `hook-check` passed green the whole time because it only verifies prerequisites (uv, jq, DB, script present), not the loop itself. The failure was invisible enough that the design was nearly abandoned (branch `feature/simplify-and-analysis`, since archived).

The call-chain bugs are fixed (PR #3, merged as fd8448c). v0.8.0 makes the loop observable, loud on failure, and self-correcting on quality — so a silent regression of this class cannot happen again, and mined memories have to earn their place.

## Goals and success criteria

1. A wiring bug of the `--session-id` class is caught by `hook-check` on day one, not by log archaeology months later.
2. After a week of normal use, `status` shows the loop producing (outputs logged, patterns mined, memories created).
3. Mined-but-never-used memories decay instead of accumulating.
4. Zero-config: every feature works out of the box with sensible defaults; configuration is for power users.

## Non-goals (explicitly out of v0.8.0)

- `mqa` (auto-promote/salience simplification), `kmu`/`fgk.15` (summarization)
- Git forensics (`t40`, `wzts`, `odnx`, `mijf`)
- Rebrand (`ab4y`) and positioning docs
- Any md-recall fold-in (markdown corpus ingestion) — separate decision, deferred

## Design

### A. Round-trip probe in `hook-check`

`hook-check` gains a live probe, on by default (`--no-probe` to skip):

1. Write a sentinel output via `storage.log_output` with a reserved probe marker.
2. Run mining synchronously scoped to the probe input.
3. Assert a mined pattern/memory materialized.
4. Hard-delete all probe artifacts (output row, patterns, memories).

Probe rows carry the reserved marker so they are excluded from stats and health metrics even if cleanup fails mid-way; the next probe (or maintenance) removes leftovers. Probe failure reports which stage failed (log, mine, assert, cleanup) with the underlying error.

This catches wiring/call-chain bugs, which cannot self-report because the child process dies at argument parsing.

### B. Loop health in `status` and the dashboard

New `mining_runs` table (schema migration): one row per mining run — started_at, finished_at, outputs_processed, patterns_found, memories_created, error (nullable). `run_mining` currently returns stats but persists nothing; it now records its run, including runtime failures.

`status` gains a "Learning loop" section: outputs logged (24h / 7d), patterns mined, memories created from mining, timestamp of last successful run. The dashboard mining page gets a health banner driven by the same queries: green (produced within 7 days), amber (no production for 7+ days), red (recent runs erroring). No log parsing anywhere; the DB is the source of truth.

### C. Loud failure at SessionStart

The SessionStart bootstrap hook additionally checks loop health and, when the loop is stale (no successful run in 7 days) or erroring (last N=3 runs failed), prepends one line to the injected context: "memory loop hasn't produced in N days — run `memory-mcp-cli hook-check`". Rate-limited to once per day. `loop_warnings_enabled=True` default.

Layering: the probe (A) catches wiring bugs; `mining_runs` (B) catches runtime errors; staleness (B/C) catches silent absence. The Stop hook keeps its must-never-break-Claude `|| true`.

### D. Injected-memory usage tracking and utility decay (`fgk.10`)

`injection_log` records what the hot cache injected; nothing records whether it helped. At Stop time, `log-response` (which already reads the transcript tail) checks whether distinctive terms from recently-injected memories appear in the assistant's responses; on a hit, bumps `used_count` and `last_used_at`.

Heuristic v1 is deliberately conservative: normalized substring match on the memory's distinctive tokens (rare terms, identifiers), not semantic matching. Best-effort — its own failure never blocks the hook and is recorded, not swallowed.

Utility decay in the maintenance job: mining-sourced memories with `retrieved_count == 0` and `used_count == 0` after 30 days are demoted from the hot cache and have their `utility_score` floored (no new schema). Pinned memories and user-seeded memories (source != mining) are exempt. Nothing is deleted in v0.8.

### E. Recall-nudge skill in the plugin

A plugin skill that triggers on retrospective questions ("didn't we…", "last time…", "how did we…") and nudges a `recall` call before answering. One SKILL.md in the plugin; drives `retrieved_count`, which feeds D's utility signal. Ported from the md-recall design.

## Data changes

- New table `mining_runs` (migration v18).
- No changes to existing columns; D uses existing `used_count`, `last_used_at`, `utility_score`, `injection_log`.

## Testing

TDD throughout. Key cases:

- Probe: full round trip on a clean DB; zero residue after success; zero stat pollution after mid-probe failure; failure reports the failing stage.
- `mining_runs`: recorded on success and on runtime error; migration test.
- Staleness thresholds: green/amber/red boundaries; SessionStart warning fires once per day and only when warranted.
- Auto-mark: fixture transcripts covering true positives, near-miss false positives, and injected-but-unused; never blocks `log-response`.
- Decay: respects pins, exempts non-mining sources, only demotes past the threshold.
- Skill file: plugin structure validation.

## Error handling

Probe and auto-mark are best-effort at runtime: failures report crisply (CLI output / `mining_runs.error` / hook log) but never corrupt stats or break hooks. SessionStart warning degrades to silence on any internal error — a broken warning system must not break session start.

## Accepted defaults (revisit only with evidence)

- Staleness: 7 days. Decay: 30 days. Error streak for red/warning: 3 runs.
- Auto-mark heuristic v1: distinctive-token substring match.
